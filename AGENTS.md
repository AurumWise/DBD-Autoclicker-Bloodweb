# DBD Autoclicker Bloodweb - Agent Instructions

## Цель проекта

DBD Autoclicker Bloodweb - внешний Windows-инструмент для помощи в прокачке Bloodweb в Dead by Daylight.

Программа работает только через:

- скриншоты экрана;
- локальное компьютерное зрение;
- собственный UI и overlay;
- обычные движения и клики мыши.

Запрещено читать память игры, внедрять DLL, обходить античит, менять файлы игры, использовать игровые API или сетевой трафик игры.

## Текущий стек

- Python 3.11+.
- `tkinter` - overlay и Python-side окна/канвас для сетки, центра и выбора узлов.
- Electron - основной пользовательский интерфейс.
- `opencv-python`, `numpy` - распознавание изображений.
- `mss` - скриншоты.
- `pynput` - движение и клики мышью.

Рабочие точки входа:

- Electron: `electron/dist/main.js` -> `app.electron_backend`.
- Main launcher: `Run_DBD_Autoclicker_Bloodweb.vbs` -> `npm.cmd run start:electron`.
- Compatibility launcher: `Run_DBD_Bloodweb_Bot.vbs` -> `Run_DBD_Autoclicker_Bloodweb.vbs`.
- Python backend: `python -m app.electron_backend`.

Не переписывать UI на PySide/PyQt без отдельного запроса. Не заменять внешний CV/autoclick-подход на способы взаимодействия с игрой напрямую.

## Основной сценарий

1. Пользователь открывает Bloodweb в игре.
2. Запускает программу.
3. Настраивает центр Bloodweb.
4. Нажимает `Паутина` и вручную совмещает сетку с Bloodweb.
5. Выбирает шаблон.
6. При необходимости добавляет скрины эталонов через режим `Скрин`.
7. Нажимает `Тест`, чтобы увидеть очередь кликов без нажатий.
8. Нажимает `Start`, чтобы программа кликала по очереди.

## UI

Интерфейс должен оставаться простым, в одном основном окне:

- настройка/подтверждение центра;
- показать/скрыть паутину;
- добавление скрина эталона;
- статичный список `Приоритет`;
- выбор текущего шаблона;
- список элементов шаблона;
- перенос элемента `В приоритет` и обратно `В шаблон`;
- удаление и переименование элементов с подтверждением;
- уровень доверия;
- настройки таймингов и мыши;
- выбор языка `ru/en`;
- `Start`, `Stop`, `Тест`;
- очередь кликов;
- следующий клик;
- окно обучения.

Не добавлять dashboard, мастера и большие панели настроек без явного запроса. Вкладки настроек уже являются частью текущего Electron UI; новые вкладки добавлять только если это явно требуется.

Окно настроек должно быть `always-on-top`. Его не нужно скрывать при `Тест` или `Start`. ROI, которые попадают под окно программы, должны игнорироваться при распознавании.

## Центр Bloodweb

Центр является защитным якорем автоклика.

Настройка центра:

- открывает overlay с прозрачным фоном;
- показывает круг, крестик в центре и вписанный шестиугольник;
- шестиугольник при настройке центра вписан в круг;
- можно перетаскивать область центра;
- можно масштабировать центральный узел мышью;
- сброс центра возвращает центр и размер узла к маленькому базовому размеру;
- первичный масштаб паутины берется из размера настроенного центра, но дальнейший пользовательский масштаб паутины сохраняется отдельно.

Перед автокликом центр должен быть подтвержден. Если центр не подтвержден, `Start` блокируется.

При `Start`, если центр временно пропал:

- клики блокируются;
- статус становится `Пауза`;
- программа ждет восстановления центра;
- если центр вернулся, старая очередь сбрасывается, делается новый скрин, строится новая очередь, работа продолжается;
- если центр не вернулся за `center_lost_timeout_seconds`, срабатывает остановка;
- `Stop`, `F8` и ручное движение мыши всегда прерывают ожидание.

Информация технической проверки центра не должна выводиться в основной UI как ошибка или лог-спам.

## Сетка и скрины эталонов

Сетка задает предполагаемые позиции узлов Bloodweb. Программа не должна искать Bloodweb по всему экрану.

Режим `Скрин`:

- открывает overlay сетки узлов;
- показывает контуры узлов;
- узлы имеют прозрачную заливку, чтобы клик по узлу не прокликивался в игру;
- при наведении узел подсвечивается заливкой;
- по клику на узел делается скрин ROI этого узла;
- после выбора узла открывается окно с полем имени и кнопками `OK` / `Отмена`;
- кнопка `Скрин` на время режима становится кнопкой отмены;
- после сохранения или отмены overlay скрина и обычная паутина не должны оставаться видимыми.

Скрин сохраняется как оригинальный PNG-эталон текущего шаблона.

Внутри распознавания применяется `crop_ratio = 0.75`: сравнивается центральная часть эталона и ROI, но match-размер сохраняется по размеру оригинального PNG.

## Геометрия паутины

Позиции узлов берутся из `app.grid.geometry.RING_ANGLES`.

Текущая схема:

- внутренний круг: 6 узлов, без изменений;
- средний круг: 12 узлов с явными углами вокруг верхней позиции;
- внешний круг: 12 узлов с явными углами вокруг верхней позиции.

Electron overlay и Python backend должны использовать одну общую геометрию.

Масштабирование:

- можно масштабировать всю паутину колесом мыши;
- можно масштабировать каждый круг отдельно;
- каждый внутренний круг не должен становиться больше следующего наружного;
- сброс размера паутины не должен сбрасывать центр и размер узла.

## Шаблоны и приоритет

Правило: одна картинка = одна запись = один список.

Запись может находиться либо:

- в `priority_items`;
- либо в одном выбранном шаблоне.

`В приоритет` переносит запись из текущего шаблона в приоритет.

`В шаблон` переносит запись из приоритета в текущий выбранный шаблон.

Это перенос, не копирование.

`Приоритет` всегда ищется раньше текущего шаблона.

Порядок списка важен: сверху выше приоритет.

При удалении элемента из приоритета или шаблона нужно подтверждение с крупным фото, названием элемента и кнопками `Удалить` / `Отмена`.

## Распознавание

Распознавание идет только внутри ROI узлов сетки.

Matcher должен искать именно картинку эталона в области ROI, а не просто белые контуры. Допустимы небольшие сдвиги, масштаб и частичные отличия.

Перед каждым скрином мышь уводится в `SCREENSHOT_MOUSE_PARKING_POSITION`, сейчас это `(0, 0)`, затем выдерживается настраиваемая задержка.

Sample и ROI нельзя сравнивать как raw-картинки. Перед сравнением оба проходят одинаковую нормализацию:

- центральный crop;
- resize к размеру конкретного эталона, а не к фиксированному `96x96`;
- percentile stretching;
- CLAHE;
- gamma-варианты для темных/светлых изображений.

Технические normalized/light/dark варианты не показываются пользователю как отдельные элементы.

Для новых пользовательских скринов технические варианты создаются сразу при сохранении PNG. Оригинал остается единственной картинкой, которую видит пользователь. Варианты хранятся рядом в скрытой папке `db/user_templates/.variants_size_v2/`.

Элемент в `db/search_templates.json` должен хранить метаданные размера:

- `image_width`;
- `image_height`;
- `match_size`;
- `capture_roi_size`;
- `crop_ratio`.

Старые записи без этих полей обогащаются из PNG при запуске.

Очередь строится так:

1. взять `priority_items`;
2. затем элементы текущего шаблона;
3. искать по порядку;
4. один элемент может встретиться несколько раз в одной паутине, для одного элемента оставляется дальний от центра узел;
5. если ничего не найдено, очередь показывает клик в центр.

`Тест` строит и показывает очередь, но не кликает.

`Start` делает один скрин, строит очередь, кликает найденные элементы по этой очереди, затем кликает центр, ждет таймер `after_center_delay_seconds` и только потом делает новый скрин.

Если во время `Start` центр был потерян и потом восстановился, старая очередь не продолжается: программа делает новый скрин и строит новую очередь.

## Мышь и остановка

Если пользователь реально сдвинул мышь во время автоклика, программа должна сразу остановиться со статусом `Пауза: мышь сдвинули`.

Не считать программные движения бота ручным сдвигом:

- во время `run_programmatic_mouse_action` включается `programmatic_mouse_action_active`;
- после клика ожидаемая позиция обновляется по фактической позиции Windows;
- порог ручного сдвига задается `mouse_move_tolerance_pixels`.

Поведение мыши при кликах:

- перед скрином мышь уводится в `(0, 0)`;
- пауза перед кликом проходит на парковке `(0, 0)`;
- мышь выходит на узел или центр только на время самого клика;
- после клика мышь сразу возвращается на парковку `(0, 0)`;
- пауза между кликами проходит на парковке.

Статусы:

- `Работа` - автоклик активен;
- `Пауза` - ожидание центра или ручной сдвиг мыши;
- `Стоп` - остановлено пользователем, F8 или таймаутом защиты;
- `Ошибка` - ошибка скриншота, распознавания или неожиданная ошибка.

## Хранение данных

Основные данные:

- `db/search_templates.json` - шаблоны, приоритет, пути к пользовательским скринам и метаданные размеров;
- `db/user_templates/` - оригинальные PNG-эталоны пользователя;
- `db/user_templates/.variants_size_v2/` - технические `.npy` варианты для matcher;
- `db/grid_templates.json` - сохраненная сетка;
- `db/ui_state.json` - позиция окна, язык UI, пороги, скорости и таймеры;
- `db/center_anchor.png` и `db/center_anchor.json` - эталон центра;
- `Icons/` и `db/*.json` - каталог игровых данных.

При удалении записи PNG удаляется только если на него больше нет ссылок.

При загрузке битые записи без существующего PNG не должны возвращаться в UI.

## Скорости и таймеры

Пользователь может менять и сохранять:

- задержку перед каждым кликом;
- задержку между кликами;
- удержание обычного клика;
- удержание клика в центр;
- задержку после клика в центр до нового скрина;
- задержку после отвода мыши перед скрином;
- таймер потери центра `center_lost_timeout_seconds`;
- интервал проверки ручного сдвига мыши;
- порог ручного сдвига мыши в пикселях;
- уровень доверия;
- порог доверия центра.

Эти значения сохраняются в `db/ui_state.json`.
