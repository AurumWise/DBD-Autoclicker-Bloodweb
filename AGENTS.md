# DBD Bloodweb Bot - Agent Instructions

## Цель проекта

DBD Bloodweb Bot - внешний Windows-инструмент для помощи в прокачке Bloodweb в Dead by Daylight.

Программа работает только через:

- скриншоты экрана;
- локальное компьютерное зрение;
- собственный UI и overlay;
- обычные движения и клики мыши.

Запрещено читать память игры, внедрять DLL, обходить античит, менять файлы игры, использовать игровые API или сетевой трафик.

## Текущий стек

- Python 3.11+.
- `tkinter` - текущий UI и overlay.
- `opencv-python`, `numpy` - распознавание изображений.
- `mss` - скриншоты.
- `pynput` - движение и клики мышью.

Не переписывать UI на PySide/PyQt без отдельного запроса. Рабочая точка входа: `main.py -> app.ui.main_window`.

## Основной сценарий

1. Пользователь открывает Bloodweb в игре.
2. Запускает программу.
3. Нажимает `Показать сетку` и вручную совмещает сетку с Bloodweb.
4. Выбирает шаблон.
5. При необходимости добавляет скрины эталонов.
6. Нажимает `Тест`, чтобы увидеть очередь кликов без нажатий.
7. Нажимает `Start`, чтобы программа кликала по очереди.

## UI

Интерфейс должен оставаться простым, в одном окне:

- показать/скрыть сетку;
- статичный список `Приоритет`;
- выбор текущего шаблона;
- список элементов шаблона;
- перенос элемента `В приоритет` и обратно `В шаблон`;
- добавление скрина;
- уровень доверия;
- настройки скорости и таймеров;
- `Start`, `Stop`, `Тест`;
- очередь кликов;
- следующий клик.

Не добавлять dashboard, вкладки, мастера и большие панели настроек без явного запроса.

Окно настроек должно быть `always-on-top`. Его не нужно скрывать при `Тест` или `Start`. ROI, которые попадают под окно программы, должны игнорироваться при распознавании.

## Сетка и скрины эталонов

Сетка задает предполагаемые позиции узлов Bloodweb. Программа не должна искать Bloodweb по всему экрану.

Рамка добавления скрина:

- появляется только в режиме добавления;
- перемещается мышью;
- не масштабируется пользователем;
- сейчас равна размеру ROI одного узла (`CAPTURE_BOX_ROI_RATIO = 1.0`);
- скрин сохраняется как оригинальный PNG-эталон текущего шаблона.

Внутри распознавания применяется `crop_ratio = 0.75`: сравнивается центральная часть эталона и ROI, но match-размер сохраняется по размеру оригинального PNG.

## Шаблоны и приоритет

Правило: одна картинка = одна запись = один список.

Запись может находиться либо:

- в `priority_items`;
- либо в одном выбранном шаблоне.

`В приоритет` переносит запись из текущего шаблона в приоритет.

`В шаблон` переносит запись из приоритета в текущий выбранный шаблон.

Это перенос, не копирование.

`Приоритет` всегда ищется раньше текущего шаблона.

Порядок списка важен: сверху выше приоритет.

## Распознавание

Распознавание идет только внутри ROI узлов сетки.

Matcher должен искать именно картинку эталона в области ROI, а не просто белые контуры. Допустимы небольшие сдвиги, масштаб и частичные отличия.

Перед каждым скрином мышь уводится в `SCREENSHOT_MOUSE_PARKING_POSITION`, сейчас это `(0, 0)`, затем выдерживается настраиваемая задержка.

Sample и ROI нельзя сравнивать как raw-картинки. Перед сравнением оба проходят одинаковую нормализацию:

- центральный crop;
- resize к размеру конкретного эталона, а не к фиксированному `96x96`;
- percentile stretching;
- CLAHE;
- gamma-варианты для темных/светлых изображений.

Технические normalized/light/dark варианты не показываются пользователю как отдельные элементы.

Для новых пользовательских скринов технические варианты создаются сразу при сохранении PNG. Оригинал остается единственной картинкой, которую видит пользователь. Варианты хранятся рядом в скрытой папке `db/user_templates/.variants_size_v2/`.

Элемент в `db/search_templates.json` должен хранить метаданные размера:

- `image_width`;
- `image_height`;
- `match_size`;
- `capture_roi_size`;
- `crop_ratio`.

Старые записи без этих полей обогащаются из PNG при запуске.

Очередь строится так:

1. взять `priority_items`;
2. затем элементы текущего шаблона;
3. искать по порядку;
4. один элемент может встретиться несколько раз в одной паутине, для одного элемента оставляется дальний от центра узел;
5. если ничего не найдено, очередь показывает клик в центр.

`Тест` строит и показывает очередь, но не кликает.

`Start` делает один скрин, строит очередь, кликает найденные элементы по этой очереди, затем кликает центр, ждет таймер `after_center_delay_seconds` и только потом делает новый скрин.

## Мышь и остановка

Если пользователь реально сдвинул мышь во время автоклика, программа должна сразу остановиться со статусом `Пауза: мышь сдвинули`.

Не считать программные движения бота ручным сдвигом:

- во время `run_programmatic_mouse_action` включается `programmatic_mouse_action_active`;
- после клика ожидаемая позиция обновляется по фактической позиции Windows;
- порог ручного сдвига задается `mouse_move_tolerance_pixels`.

## Хранение данных

Основные данные:

- `db/search_templates.json` - шаблоны, приоритет, пути к пользовательским скринам и метаданные размеров;
- `db/user_templates/` - оригинальные PNG-эталоны пользователя;
- `db/user_templates/.variants_size_v2/` - технические `.npy` варианты для matcher;
- `db/grid_templates.json` - сохраненная сетка;
- `db/ui_state.json` - позиция окна, порог доверия, скорости и таймеры;
- `Icons/` и `db/*.json` - каталог игровых данных.

При удалении записи PNG удаляется только если на него больше нет ссылок.

При загрузке битые записи без существующего PNG не должны возвращаться в UI.

## Скорости и таймеры

Пользователь может менять и сохранять:

- задержку перед каждым кликом;
- задержку между кликами;
- удержание обычного клика;
- удержание клика в центр;
- задержку после клика в центр до нового скрина;
- задержку после отвода мыши перед скрином;
- интервал проверки ручного сдвига мыши;
- порог ручного сдвига мыши в пикселях;
- уровень доверия.

Эти значения сохраняются в `db/ui_state.json`.
